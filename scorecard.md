# Skill Scorecard

这个文件用于把 skill 审查从“感觉不错”变成“可以打分”。

你可以用它来：

- 审查一个现有 skill
- 比较两个 skill 谁更成熟
- 在重写前后做对比
- 给团队建立统一的 skill 质量标准

---

## 使用方式

每个维度按 0 到 5 分评分：

- `0` = 完全没有
- `1` = 提到了，但几乎不起作用
- `2` = 有初步结构，但比较弱
- `3` = 基本合格，可用
- `4` = 明显扎实，能稳定约束行为
- `5` = 非常强，可作为其他 skill 的参考样板

建议：

- 先逐项评分
- 再看总分
- 最后写“最该优先修的 3 个点”

## 一、Identity（身份边界）

**问题：**

- skill 有没有清楚说明自己是什么？
- skill 有没有清楚说明自己不是什么？
- 它是否容易膨胀成“万能 skill”？

**高分特征：**

- 有明确角色定义
- 有反向定义（不负责什么）
- 不会轻易越界到相邻任务

## 二、Trigger Discoverability（触发可发现性）

**问题：**

- description 是否真的围绕触发条件写？
- 用户说的话，是否容易命中这个 skill？
- skill 是否有足够的症状词、场景词、任务词？

**高分特征：**

- description 很容易检索到
- 适合的触发场景非常明确
- 不会因为太抽象而“找不到”

**评分：** `/5`

## 三、Scope（范围定义）

**问题：**

- skill 有没有写清楚适用场景？
- 有没有写清楚不适用场景？
- 用户是否知道什么时候该换别的 skill？

**高分特征：**

- `Use when` 明确
- `Do not use when` 明确
- 与邻近 skill 的边界清楚

## 四、Ordered Constraints（约束是否写进流程）

**高分特征：**

- 先做什么、后做什么写得清楚
- 重要限制在步骤里可执行
- agent 不容易只读原则不读流程

- 明确 fallback

### 参考解释

- `0-20`
  - 非常弱，更多只是提示词，不足以称为稳定 skill

- `21-35`
  - 初步可用，但容易失控，边界和输出都不够稳

- `36-45`
  - 合格 skill，有明显结构，但仍有增强空间

- `46-54`
  - 强 skill，具备比较成熟的边界、约束与输出设计

- `55-60`
  - 样板级 skill，值得被其他 skill 学习和复用
