# Review Example: A Weak Skill

这个示例不是针对某个真实开源仓库，而是一个典型“看起来像 skill，但其实不够听话”的结构化反例。

它的作用是帮助大家识别：

- 什么样的 skill 只是“像个 prompt”
- 什么样的问题会导致 skill 失控
- 为什么 boundary / workflow / fallback / output contract 很重要

---

## 假设的原始 skill

下面是一个典型弱 skill 的风格：

> 你是一个很强的产品设计专家和前端专家。  
> 当用户需要页面、设计稿、原型、交互、网站、组件、文档时，你都可以帮他快速完成。  
> 请尽量做得高级、专业、好看。  
> 你可以灵活判断，不需要太拘泥。  
> 输出时尽量完整。

---

## Skill Type

这类 skill 通常表面上像“设计 skill”，但实际上是：

- 角色混乱的通用生成器

它没有真正稳定的身份。

---

## Primary Problem

它的核心问题不是“不够聪明”，而是：

**没有边界，也没有约束执行机制。**

---

## 按 scorecard 看，它为什么弱

### 1. Identity 很弱

问题：

- 既说自己是设计专家
- 又说自己是前端专家
- 又能做页面、原型、网站、组件、文档

结果：

- 角色膨胀
- 什么都能接
- 什么都可能做歪

### 2. Trigger Discoverability 很弱

问题：

- 触发条件太泛
- 几乎任何和“设计 / 页面 / 文档”有关的话都能触发

结果：

- skill 被误用概率很高

### 3. Scope 很弱

问题：

- 没有写适用场景
- 没有写不适用场景

结果：

- 用户不知道什么时候该换 skill
- agent 也不会主动收敛范围

### 4. Ordered Constraints 几乎没有

问题：

- 没有明确先做什么后做什么
- “尽量高级、专业、好看”只是审美口号，不是 workflow

结果：

- agent 只能凭默认先验自由发挥

### 5. Gates 没有

问题：

- 没有确认门
- 没有验证门
- 没有完成门

结果：

- 容易过早结束
- 容易在需求还没对齐时就进入生成

### 6. Output Contract 很弱

问题：

- 只说“尽量完整”
- 没有定义输出结构

结果：

- 每次输出风格都可能变
- 不利于复核和协作

### 7. Fallback 没有

问题：

- 信息不足时没有安全退化路径

结果：

- 最常见行为就是“硬编一个看起来不错的东西”

### 8. Evidence Before Output 没有

问题：

- 没有要求先读上下文、品牌、代码、文档

结果：

- 非常容易做出 generic output

### 9. Validation Story 没有

问题：

- 没有任何检查方式

结果：

- 输出是否靠谱只能靠感觉

### 10. Anti-Slop 能力很弱

问题：

- 没有禁止默认套路

结果：

- 很容易滑向“紫渐变 + 圆角卡片 + 通用大标题”的平均化输出

---

## 为什么这类 skill 常常“不听话”

因为它把最重要的东西都留在“自由判断”里：

- 什么时候该收敛
- 什么时候该停
- 什么时候该问
- 什么时候该 fallback
- 最终结果长什么样

一旦这些没有结构化，agent 就会用自己的默认先验补全。

这就是 skill 失控的根本原因。

---

## 这种 skill 应该怎么改

最优先的修法通常是：

1. 先给角色做 `Role Lock`
2. 再给任务做 `Scope Split`
3. 把关键约束写进 workflow
4. 补 fallback
5. 补 output contract
6. 补 negative list

---

## 一句话结论

这类 skill 最大的问题不是“写得不够多”，而是：

**没有把判断变成结构。**

所以它更像一个泛 prompt，而不是一个稳定 skill。
